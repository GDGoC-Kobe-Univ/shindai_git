# スナップショット
### 1. スナップショットの考え方
従来のバージョン管理システム（SVNなど）は、ファイルごとの「変更内容（デルタ）」を積み重ねて管理する。一方、Gitはコミットするたびに、その時の全ファイルのコピーをひとまとめにした「スナップショット」を作成する。
### 2. スナップショットの仕組み
効率化のため、Gitは毎回すべてのファイルを律儀に物理コピーしているわけではない。
- **変更がないファイル:** 前回のスナップショットにある同じファイルへの**リンク（参照）**を格納する。
- **変更があるファイル:** 新しく作成されたファイルの実体を保存し、そのスナップショットから参照する。
これにより、ストレージ容量を節約しつつ、特定のコミット時点の状態を瞬時に復元できる。
### 3. メリット
- **高速な切り替え:** 過去のバージョンに戻る際、差分を計算し直す必要がなく、スナップショットを展開するだけで済む。
- **データの整合性:** ファイルの一部が壊れても、スナップショット単位で管理されているため、不整合が起きにくい。
### 4. まとめ表

| **特徴**       | **デルタベース (SVN等)**     | **スナップショットベース (Git)** |
| ------------ | --------------------- | --------------------- |
| **管理単位**     | ファイルごとの変更履歴           | プロジェクト全体の瞬間状態         |
| **データ保持**    | 基本ファイル + 差分1 + 差分2... | その時点の全ファイル（またはリンク）    |
| **ブランチ切り替え** | 差分の計算が必要で遅い           | 参照を切り替えるだけで高速         |
